# KPI / Показатели: верхнеуровневая вкладка

## Контекст

В системе уже есть два рабочих контура:

- `Заявки клиентов` / коммерческий контур
- `RFQ и закупка` / закупочный контур

На текущем этапе не хватает отдельной управленческой вкладки верхнего уровня, где можно оценивать:

- результативность продавцов
- результативность закупщиков
- узкие места процесса
- загрузку и скорость работы сотрудников

Такая вкладка не должна находиться внутри `Каталогов`, потому что это не справочник, а управленческий слой над бизнес-процессом.

Рекомендуемое название вкладки:

- `Показатели`

Допустимый альтернативный вариант:

- `Показатели и KPI`

## Что нужно сделать

## Логика KPI

## KPI продавца

Продавец отвечает за:

- заявку клиента
- КП
- контракт

### Основные показатели

#### Объем

- новых заявок за период
- выпущенных КП за период
- подписанных контрактов за период
- сумма подписанных контрактов

#### Скорость

- среднее время от заявки до первого КП
- среднее время от первого КП до подписания контракта
- среднее время от заявки до контракта

#### Конверсия

- заявка -> КП
- КП -> контракт
- заявка -> контракт

#### Потери и зависание

- количество заявок без КП
- количество заявок в затяжном торге
- количество заявок, закрытых без контракта

## KPI закупщика

Закупщик отвечает за:

- RFQ
- ответы поставщиков
- покрытие
- сценарии
- логистику
- экономику
- выбор
- PO
- supplier quality по своему закупочному контуру

### Основные показатели

#### Объем

- RFQ за период
- утвержденных выборов (`selection`)
- созданных PO
- количество привлеченных поставщиков

#### Скорость

- среднее время от выпуска в закупку до `selection`
- среднее время от `selection` до первого `PO`
- среднее время от подписанного контракта до первого `PO`

#### Качество закупочного контура

- доля строк с полным покрытием
- доля строк с ценой
- доля сценариев без missing origin
- доля сценариев без missing weight
- доля кейсов без supplier incidents

#### Исполнение

- количество задержек поставщиков по его закупкам
- количество рекламаций
- количество кейсов с `closed_with_issues`

## KPI начальника / руководителя

Нужна сводная панель:

- сравнение продавцов
- сравнение закупщиков
- узкие места по этапам
- воронка заявка -> КП -> контракт -> PO
- инциденты по поставщикам
- просрочки
- нагрузка по сотрудникам

## Чего не хватает для сильного KPI-модуля

## 1. Не все milestone-датировки зафиксированы явно

Для точных KPI желательно иметь отдельные milestone-метки:

- дата первого КП
- дата подписания контракта
- дата первого PO
- дата завершения контракта

Часть этого сейчас можно выводить косвенно, но лучше иметь явные поля или хорошо определенные derived rules.

## 2. Не везде зафиксирован owner как источник ответственности

Для KPI нельзя опираться на "кто последний редактировал".
Нужен четкий owner по этапу:

- продавец по заявке
- закупщик по RFQ
- ответственный по контракту
- ответственный по PO

Часть owner-полей уже есть, но их надо пройти и проверить на полноту использования.

## 3. Не хватает причин потери / закрытия без результата

Для воронки важно понимать:

- почему заявка не дошла до контракта
- почему контракт не дошел до PO
- почему кейс ушел в проблемы

## Рекомендуемые изменения для следующего этапа

### Таблица целей KPI

```sql
CREATE TABLE kpi_targets (
  id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  role_id INT NULL,
  user_id INT NULL,
  period_type VARCHAR(16) NOT NULL,
  period_start DATE NOT NULL,
  period_end DATE NOT NULL,
  metric_code VARCHAR(64) NOT NULL,
  target_value DECIMAL(18,4) NOT NULL,
  note TEXT NULL,
  created_by_user_id INT NOT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_kpi_targets_period (period_start, period_end),
  KEY idx_kpi_targets_role_user (role_id, user_id),
  KEY idx_kpi_targets_metric (metric_code)
);
```

Это уже даст возможность сравнивать:

- факт
- план
