# RFQ / Client Request / Contract / PO Audit Plan

Дата: 2026-03-16

## 1. Цель документа

Этот документ фиксирует текущее понимание сквозного процесса:

`Заявка клиента -> RFQ -> Покрытие -> Сценарий -> Логистика -> Экономика -> Выбор закупки -> КП -> rev КП -> Контракт -> PO -> Исполнение -> Supplier Quality`

Документ нужен как handoff для продолжения работы в новом чате без потери контекста.

Здесь зафиксированы:

- что уже сделано и работает логично;
- где остаются реальные логические разрывы;
- какие бизнес-решения уже приняты;
- какие вопросы еще открыты;
- в каком порядке надо доводить систему до более зрелого production-процесса.

## 2. Что уже сделано и считается правильным

### 2.1. RFQ и экономика

- `Incoterms` и `incoterms_place` больше не читаются как полностью live-данные из response line, а снапшотятся в `rfq_coverage_option_lines`.
- `origin_country` больше не подменяется `supplier_country` в customs/economics-логике.
- warnings по логистике и экономике усилены:
  - `mixed_origin_countries`
  - `mixed_incoterms`
  - `mixed_incoterms_places`
  - `missing_customs_data`
  - `whole_uom_mismatch`
  - `multiple_whole_lines`
- FX-провайдер обновлен так, чтобы нормально поддерживать основные валюты, включая `RUB`.

### 2.2. Procurement baseline

- `Selection` теперь является корректной точкой фиксации procurement baseline.
- В `selection_lines` сохраняется `supplier_public_code_snapshot`, чтобы seller-side не зависел от live-изменения supplier codes.

### 2.3. Seller-side

- Из `selection` создается `sales quote`.
- Продавец работает через `sales_quote_revisions`, а не через изменение procurement-слоя.
- В `sales_quote_lines` добавлен `line_status` (`active` / `excluded`), чтобы rev КП могла отражать торг клиента по составу.

### 2.4. Contract and PO

- `client_contracts` теперь могут ссылаться на конкретную `sales_quote_revision_id`.
- `PO` создается не по всему исходному `selection`, а по подписанной коммерческой ревизии:
  - исключенные строки не попадают в `PO`;
  - количество берется из утвержденной коммерческой rev.

### 2.5. Supplier quality

- Supplier quality events теперь можно создавать не только из карточки поставщика, но и из `PO`.
- Событие качества может связываться с:
  - `supplier_purchase_order_id`
  - `supplier_purchase_order_line_id`
  - `rfq_response_line_id`
  - `selection_id`
  - `selection_line_id`
  - `sales_quote_id`
  - `sales_quote_line_id`
  - `oem_part_id`
- Появилась возможность обновлять и закрывать событие качества.

## 3. Уже принятые бизнес-решения

### 3.1. По контрактам

Принято решение:

- по одной клиентской заявке должен быть один финальный контракт;
- заявка клиента в итоге закрывается контрактом.

Это означает, что модель "несколько равноправных signed-контрактов на один и тот же процесс" не соответствует целевому бизнес-потоку.

Следствие:

- в системе нужно постепенно двигаться к модели одного финального активного контракта на один client request flow;
- если есть технически несколько контрактов, это должно трактоваться как исторические/ошибочные/промежуточные состояния, а не как нормальная целевая модель.

### 3.2. По supplier quality

Принято решение:

- supplier quality events должны со временем влиять на supplier scoring;
- цель модуля не только журнал инцидентов, но и накопление статистики надежности поставщиков.

Следствие:

- supplier quality должен стать одним из источников для `reliability_rating`, `risk_level`, future supplier ranking.

## 4. Открытый вопрос: что такое approved procurement handoff

Пояснение простыми словами.

Сейчас после подписанного контракта закупка понимает, что можно выпускать `PO`, потому что есть связка:

- `signed contract`
- `sales_quote_revision`
- `PO autofill from signed revision`

Это уже работает.

Но архитектурно есть два варианта:

### Вариант A. Ничего нового не вводить

Считать, что роль "утвержденного handoff в закупку" уже выполняет связка:

- подписанный контракт;
- конкретная `sales_quote_revision`;
- логика `PO`, которая берет только approved active lines этой rev.

Плюсы:

- проще;
- меньше новых сущностей;
- уже частично реализовано.

Минусы:

- процессное значение "что именно разрешено запускать в закупку" живет не в отдельном объекте, а выводится косвенно;
- сложнее аудит и объяснение состояния;
- сложнее в будущем строить явные workflow-экраны для handoff.

### Вариант B. Ввести отдельный объект approved procurement handoff

Например:

- `approved_fulfillment_snapshot`
- или `procurement_release`

Это был бы отдельный зафиксированный срез, созданный после `contract signed`, в котором явно записано:

- какая rev КП утверждена;
- какие строки реально идут в исполнение;
- какие количества утверждены;
- какой procurement scope передан закупщику.

Плюсы:

- процесс становится более прозрачным;
- лучше аудит и контроль;
- проще разделить коммерческий слой и execution layer.

Минусы:

- дополнительная сложность модели;
- нужна новая таблица/сущность и новый UI.

### Текущая рекомендация

На текущем этапе можно жить без нового объекта, если:

- в системе строго один финальный контракт;
- `contract -> sales_quote_revision -> active lines` однозначно определяют procurement scope.

То есть сейчас отдельный объект handoff не обязателен.

Но если позже появится:

- сложный контрактный workflow,
- частичное исполнение по этапам,
- несколько коммерческих итераций на один заказ,
- формальный release management,

тогда отдельный `approved procurement handoff` станет оправданным.

## 5. Самые важные логические разрывы на сегодня

Ниже перечислены реальные риски, а не косметика.

### 5.1. Критично: добавление строки в rev КП из чужой client request revision

Сейчас backend допускает сценарий, где в `sales_quote_revision` можно добавить строку, которая не относится к той же клиентской ревизии, что и сам quote.

Почему это опасно:

- seller-side rev может потерять логическую связь с исходной заявкой;
- контракт и `PO` дальше будут опираться на коммерческий состав, который не соответствует исходной ревизии спроса.

Это один из самых опасных оставшихся разрывов.

### 5.2. Критично: выбор основания для PO через "последний signed contract"

Сейчас `PO` берет коммерческий baseline как "последний signed contract" по `selection`.

Это допустимое упрощение, но уже не соответствует принятому бизнес-решению "один финальный контракт на заявку".

Если в данных вдруг окажется несколько signed-контрактов, система выберет один по дате/ID, а не по явно утвержденному бизнес-правилу.

### 5.3. Высокий риск: глобальный экран PO может показывать не тот upstream context

На общей странице `purchase orders` строка `PO` привязывается к `selection_lines` через `rfq_response_line_id` недостаточно строго.

Если одна и та же `rfq_response_line` исторически попала в несколько `selection`, общий экран может показать неправильный upstream context.

Это опасно потому, что:

- качество,
- анализ исполнения,
- трассировка back to RFQ

могут опираться на неточный контекст.

### 5.4. Высокий риск: procurement baseline после Selection еще не полностью самодостаточен

В `selection_lines` уже снапшотятся деньги и supplier snapshots, но не все execution-critical поля замораживаются полностью.

Например:

- `route_type` сейчас сохраняется как `null`;
- `incoterms` / `incoterms_place` не доведены до явного selection snapshot;
- execution layer все еще частично зависит от ручного ввода в `PO`.

Следствие:

- handoff в закупку не является полностью самодостаточным слепком.

### 5.5. Средний риск: статусы процесса не защищены как state machine

Статусы `sales quote` и `contract` сейчас не оформлены как строгий автомат переходов.

Следствие:

- UI ведет себя правильно;
- но backend не защищает процесс достаточно жестко от невалидных переходов, если статус будет изменен внешним кодом или ручным вызовом API.

### 5.6. Средний риск: supplier quality пока не влияет на supplier scoring

Модуль уже умеет хорошо фиксировать инциденты, но пока еще не превращает их в управленческий эффект для будущих RFQ.

То есть статистика копится, но decision loop еще не замкнут.

## 6. Вопрос о закрытии контракта как выполненного

Это отдельный важный бизнес-вопрос.

Текущая логика:

- контракт подписан -> можно выпускать `PO`.

Но непонятно, когда контракт считается успешно исполненным.

### Возможные трактовки

#### Вариант 1. Контракт выполнен, когда размещены PO

Это слишком рано.

Почему плохо:

- размещение `PO` не означает фактическое исполнение поставки;
- поставщики могут сорвать сроки, поставить не все, поставить с дефектом.

#### Вариант 2. Контракт выполнен, когда все supplier PO закрыты

Это уже лучше, но все еще не идеально.

Почему:

- закрытый `PO` со стороны поставщика не всегда означает, что клиент получил и принял всё без рекламаций.

#### Вариант 3. Контракт выполнен, когда:

- все утвержденные строки контракта реально обеспечены поставками;
- клиент получил товар;
- нет открытых критичных претензий/рекламаций по этому контракту;
- обязательства по объему закрыты.

Это наиболее зрелая трактовка.

### Текущая рекомендация

Сейчас разумно считать так:

- `signed` = коммерчески утвержден и открыт к исполнению;
- `completed` / `fulfilled` = все утвержденные строки исполнены и нет открытых блокирующих отклонений.

То есть контракт не должен становиться "выполненным" просто из-за размещения `PO`.

### Что это значит для системы

В будущем контрактному контуру нужен отдельный статус исполнения, например:

- `draft`
- `sent_to_client`
- `signed`
- `in_execution`
- `completed`
- `closed_with_issues` или `closed_partial` при необходимости

Сейчас это еще не реализовано и является отдельным слоем следующего этапа.

## 8. Рекомендуемый порядок внедрения

Порядок именно по риску:

1. запрет чужих строк в `rev КП`;
2. правило одного финального signed-контракта;
3. исправление глобальной трассировки `PO`;
4. доснапшот execution-полей в `selection_lines`;
5. автозаполнение `PO` из approved baseline;
6. supplier quality -> scoring;
7. lifecycle исполнения контракта.

## 9. Что не требует немедленного вмешательства

Можно пока оставить как допустимые упрощения:

- отсутствие отдельного объекта `approved procurement handoff`, если будет жестко соблюдаться модель одного финального контракта;
- сохранение терминов `RFQ`, `PO`, `rev`, `Incoterms`, `GM%` как общепринятых;
- ручное закрытие supplier quality events, если остальная трассировка уже корректна.
