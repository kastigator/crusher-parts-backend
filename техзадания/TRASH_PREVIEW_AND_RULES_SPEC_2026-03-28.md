# Preview, правила удаления и анализ пересечений в БД

Дата: 2026-03-28

Статус: аналитический spec, без изменений в коде и БД

Связанный документ:

- `TRASH_DELETE_POLICY_ANALYSIS_2026-03-28.md`

Цель документа:

- формализовать UX и backend contract для preview перед удалением
- зафиксировать правила принятия решения `trash / archive / relation_delete / forbidden`
- отдельно разобрать БД на предмет мусора и пересечений с будущей корзиной

---

## 1. Главный продуктовый принцип

Пользователь не должен нажимать одну и ту же кнопку `Удалить` для сущностей с разным смыслом.

Система должна различать 4 режима:

- `trash`
- `archive_only`
- `relation_delete`
- `forbidden`

Решение о режиме должно приниматься сервером после preview-анализа.

## 3. Что должен отображать frontend

В modal preview нужно всегда показывать 4 блока.

### 3.1. Что будет затронуто

- дочерние сущности
- локальные зависимости
- связи
- документы

Формат:

- понятные бизнес-названия
- счетчики
- группировка по смыслу, а не по внутренним таблицам

### 3.2. Что не будет затронуто

Показывать важные объекты, которые останутся:

- завершенные процессы
- исторические документы
- внешние каталожные сущности

Это снижает страх пользователя.

### 3.3. Ограничения

Показывать:

- активные процессы
- блокирующие статусы
- причины запрета
- альтернативное действие

### 3.4. Что будет дальше

Показывать:

- уйдет ли объект в корзину
- можно ли восстановить
- срок хранения
- кто имеет право на окончательное удаление

## 4. Типы preview по режимам

## 4.1. Режим `trash`

Пользователь видит:

- объект будет перемещен в корзину
- будет скрыт из обычных списков
- связанные зависимые записи также будут скрыты
- восстановление доступно

Кнопки:

- `Переместить в корзину`
- `Отмена`

## 4.2. Режим `archive_only`

Пользователь видит:

- объект участвует в исторически значимом или активном процессе
- удаление в корзину недоступно
- доступно только архивирование

Кнопки:

- `Архивировать`
- `Отмена`

## 4.3. Режим `relation_delete`

Пользователь видит:

- будет удалена только связь
- сам объект останется
- будут затронуты только локальные данные, относящиеся к этой связи

Кнопки:

- `Удалить связь`
- `Отмена`

## 4.4. Режим `forbidden`

Пользователь видит:

- удаление запрещено
- перечислены причины
- показано, какие действия нужно сначала выполнить

Кнопки:

- `Понятно`
- опционально альтернативное бизнес-действие

## 5. Правила принятия решения

Ниже указана целевая логика для server-side preview service.

## 5.1. Клиенты

Если у клиента нет активных процессов:

- mode = `trash`

Если есть активные `client_requests` или `rfqs`:

- mode = `archive_only`

Если в будущем будет введена строгая политика immutable-history и архив недопустим:

- mode = `forbidden`

Preview должен показывать:

- contacts
- billing addresses
- shipping addresses
- bank details
- equipment units
- active requests
- active RFQ

## 5.2. Поставщики

Если поставщик не участвует в активном процессе:

- mode = `trash`

Если участвует в RFQ / PO / quality:

- mode = `archive_only`

Если по политике компании поставщик с закупочной историей вообще не должен скрываться:

- mode = `forbidden`

Preview:

- contacts
- addresses
- bank details
- supplier parts
- active RFQ
- active purchase orders
- quality events

## 5.3. Client equipment units

Если нет критичного активного зависимого процесса:

- mode = `trash`

Если unit прямо нужна для незавершенного workflow:

- mode = `forbidden`

Preview:

- unit-specific overrides
- material overrides

## 5.4. OEM parts

Если пользователь удаляет OEM из контекста одной модели:

- mode = `relation_delete`

Если пользователь удаляет OEM целиком и активного использования нет:

- mode = `trash`

Если есть активное использование в незавершенных процессах:

- mode = `archive_only` или `forbidden`

Preview:

- fitments
- BOM links
- documents
- alternatives
- standard links
- supplier links
- process usage

## 5.5. Standard parts

Если standard part не участвует в активном процессе:

- mode = `trash` или `archive_only`

Если участвует:

- mode = `archive_only` или `forbidden`

Для classifier roots:

- mode = `forbidden`
- допустимы только admin-only actions

## 5.6. Supplier parts

Если supplier part не используется в active response / selection:

- mode = `trash`

Если используется:

- mode = `archive_only` или `forbidden`

Для links:

- mode = `relation_delete`

## 5.7. Client requests

Если request = draft и не создал downstream aggregate:

- mode = `trash`

Если уже есть RFQ или другой downstream-след:

- mode = `archive_only`
- пользовательская альтернатива: `cancel / close`

## 5.8. RFQ

Если RFQ = draft и не отправлен:

- mode = `trash`

Если отправлен или получил responses:

- mode = `archive_only`
- пользовательская альтернатива: `cancel / close`

## 5.9. Supplier responses

По умолчанию:

- mode = `archive_only`

Пользовательское hard delete не рекомендуется.

## 5.10. Quotes / Contracts / PO / Quality

По умолчанию:

- mode = `forbidden`

Пользовательские альтернативы:

- `cancel`
- `void`
- `close`
- `archive`

## 5.11. Users / Roles / System roots

По умолчанию:

- mode = `forbidden`

Альтернатива:

- `deactivate`
- `hide from selectors`

## 6. Что пользователь должен сделать до удаления

Для части сущностей до удаления нужно выполнить обязательные шаги.

## 6.1. Клиент

Если есть активная заявка или RFQ:

- сначала закрыть / отменить процесс
- либо архивировать клиента вместо удаления

## 6.2. Поставщик

Если поставщик участвует в активном RFQ / PO:

- сначала закрыть активные закупочные процессы
- либо архивировать поставщика

## 6.3. OEM деталь

Если деталь используется в активном процессе:

- сначала снять ее из текущего процесса или завершить процесс
- либо архивировать деталь
- либо удалить только fitment, а не всю деталь

## 6.4. Client request / RFQ

Если объект уже в процессе:

- вместо удаления пользователь должен:
  - отменить
  - закрыть
  - архивировать

## 6.5. Документы процесса

Для quote / contract / PO:

- сначала провести допустимую бизнес-операцию статуса
- удаление как корзина не должно быть доступно обычному пользователю

---

## 7. Восстановление: требования к UX

Перед восстановлением нужно делать restore preview.

Показывать:

- что восстановится
- что не восстановится
- есть ли конфликты
- что именно конфликтует

Возможные исходы:

1. полное восстановление
2. частичное восстановление
3. восстановление невозможно

Если есть конфликт уникальности или конфликт структуры:

- пользователь должен видеть понятную причину
- никакого silent failure

## 8. Требования к будущему backend service layer

Для удаления нужна не только операция `delete`, а отдельный набор сервисов:

- `buildDeletePreview(entityType, entityId, context)`
- `executeDeleteAction(previewDecision)`
- `buildRestorePreview(trashEntryId)`
- `executeRestoreAction(trashEntryId, strategy)`

Delete service не должен жить только внутри route handlers.

Причина:

- сейчас delete-логика размазана по роутам
- для корзины потребуется единый движок принятия решений и snapshot-сборки

## 9. Анализ БД на пересечение с будущей корзиной

Проведена проверка текущей схемы.

### 9.1. Что уже есть

В БД есть:

- `activity_logs`

На момент начала анализа в БД также были старые migration backup-таблицы OEM-refactor, но в рамках cleanup этого тренировочного контура они удалены как неиспользуемый мусор и больше не должны учитываться в будущей архитектуре корзины.

### 9.2. Чего нет

На текущий момент в схеме отсутствуют:

- отдельные `trash_*` таблицы
- отдельные `archive_*` таблицы
- `deleted_at`
- `deleted_by`
- `is_deleted`
- `archived_at`
- `archived_by`
- `restore_status`
- `purge_after_at`

Вывод:

- прямого пересечения с будущей корзиной сейчас почти нет
- новая корзина не конфликтует со старой trash-моделью, потому что старой trash-модели в БД нет

### 9.3. Что потенциально может создать кашу

Не прямой конфликт, но архитектурный шум создают:

1. `activity_logs` с delete-записями
2. legacy naming в логах (`original_parts`, `supplier_part_originals` и т.п.)
3. frontend/UI, где "Удаленные" означает историю delete-логов, а не настоящую корзину

### 9.4. Что НЕ является корзиной и не должно подменять ее

#### `activity_logs`

Это аудит.

Не надо:

- использовать `activity_logs` как trash storage
- использовать delete-комментарии как snapshot для восстановления

Оставить:

- как журнал действий

#### Старые migration backup-таблицы

На момент подготовки этого документа старые OEM migration backup-таблицы уже удалены из тренировочной БД как неиспользуемый мусор.

Вывод:

- они не должны учитываться в будущей модели корзины
- в рабочем контуре аналогичные таблицы можно удалять только после отдельной проверки

## 10. Что можно считать кандидатами на будущую зачистку

На этапе внедрения корзины стоит рассмотреть cleanup следующих вещей.

### 10.1. UI/API naming cleanup

После появления настоящей корзины нужно будет развести:

- `Deleted logs`
- `Trash`

Иначе пользователь увидит 2 разных сущности с похожим названием.

Нужно будет:

- переименовать текущие экраны "Удаленные", которые показывают `activity_logs`
- назвать их, например:
  - `История удалений`
  - `Журнал удалений`

### 10.2. Legacy entity naming cleanup

В логах и части кода еще встречаются legacy названия сущностей.

Их нужно нормализовать, чтобы будущая корзина не наследовала путаницу.

### 10.3. Backup tables review

В тренировочной БД старые OEM migration backup-таблицы уже удалены.

Для будущих cleanup-решений в других контурах правило остается таким:

перед удалением временных backup-таблиц нужно убедиться, что они:

1. не используются кодом и ручными скриптами
2. не нужны для rollback
3. не содержат единственную копию полезных данных

### 10.4. Delete routes refactor

После внедрения корзины часть текущих hard delete маршрутов станет legacy и должна быть переработана.

Иначе в системе будет coexistence:

- старый hard delete
- новый trash delete

Это и будет реальной кашей.

---

## 11. Что сейчас удалять нельзя

На текущем этапе не следует удалять:

- `activity_logs`
- записи из `activity_logs`
- старые delete-комментарии

Причина:

- это действующий исторический след прошлых удалений
- это не конфликтует напрямую с новой корзиной
- удаление сейчас не даст архитектурного выигрыша

---

## 12. Что можно зафиксировать как policy по чистоте системы

Чтобы не плодить мусор при внедрении корзины:

1. не использовать `activity_logs` как storage корзины
2. не добавлять параллельно еще один "deleted history" UI с тем же смыслом
3. не делать одновременно hard delete и trash delete для одной и той же пользовательской кнопки
4. не вводить `archived`, `deleted`, `hidden` без явного различия семантики
5. не трогать backup-таблицы без отдельного cleanup decision
