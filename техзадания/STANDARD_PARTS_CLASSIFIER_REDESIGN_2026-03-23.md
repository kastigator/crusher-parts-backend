- Репозиторий: `crusher-parts-backend`
- Связанный frontend: `../crusher-parts-frontend`
- Статус документа: на согласовании
- Реализация: не начата

## 1. Контекст и причина изменений

Текущий каталог стандартных деталей реализован как одна таблица `standard_parts` с фиксированным набором полей.

Сейчас это работает как плоский справочник, где для любой стандартной детали заранее заданы одни и те же атрибуты:

- `part_type`
- `designation`
- `standard_system`
- `strength_class`
- `material_spec`
- `coating`
- `thread_spec`
- `size_note`
- `uom`
- `description_ru`
- `description_en`
- `notes`
- `is_active`

Такой подход подходит только для узкого набора крепежа. Для разных классов стандартных изделий требуются разные характеристики:

- болты: резьба, длина, класс прочности, покрытие
- гайки: шаг резьбы, высота, тип гайки, класс
- шайбы: внутренний/наружный диаметр, толщина, тип
- электродвигатели: мощность, напряжение, обороты, IP, исполнение, фланец
- гидроцилиндры: ход, диаметр штока, рабочее давление, тип крепления

Итог: текущая структура каталога концептуально неверна, потому что модель данных не отражает различие типов стандартных изделий.

## 3. Текущее состояние системы

### 3.1 Как каталог устроен сейчас

`standard_parts` сейчас является одновременно:

- каталогом типов изделий
- каталогом конкретных стандартных позиций
- хранилищем характеристик
- источником отображаемого названия

То есть в одной строке смешаны:

- тип изделия
- идентичность позиции
- свойства позиции
- поисковые и экранные поля

### 3.2 Как это используется в системе

`standard_part_id` уже встроен в бизнес-процессы и используется во многих таблицах:

- `client_request_revision_items`
- `client_request_revision_item_components`
- `rfq_items`
- `rfq_item_components`
- `rfq_response_lines`
- `rfq_coverage_option_lines`
- `rfq_supplier_line_selections`
- `supplier_quality_events`
- `oem_part_standard_parts`
- `supplier_part_standard_parts`

Ключевой вывод:

- идентификатор `standard_part_id` уже стал важной бизнес-сущностью
- ломать таблицу `standard_parts` и заменять ее другой сущностью рискованно
- правильнее сохранить `standard_parts` как главную сущность позиции, но поменять внутреннюю модель атрибутов

### 3.3 Что жестко зашито

Жесткая модель зашита одновременно:

- в БД через фиксированные колонки
- в backend API через фиксированный payload
- во frontend через фиксированную форму создания и редактирования
- в поиске и таблицах через жесткий список колонок

### 3.4 Где используются атрибуты standard parts

Сейчас почти везде от standard part нужны только базовые данные:

- `id`
- `part_type`
- `designation`
- `standard_system`
- иногда описание

Большинство бизнес-процессов работают не с полным набором характеристик, а только с самим `standard_part_id` и коротким отображением.

Это хорошо для рефакторинга:

- можно менять внутреннюю структуру каталога
- при этом сохранить обратную совместимость по `standard_part_id`

## 4. Основные проблемы текущей реализации

### 4.1 Неправильная предметная модель

Разные типы стандартных изделий принудительно укладываются в одну схему полей.

### 4.2 Нет настраиваемого классификатора

Нельзя создать новый класс стандартных деталей и описать для него свой набор атрибутов без доработки кода и БД.

### 4.3 Нельзя управлять формой карточки из интерфейса

Чтобы добавить новый параметр для типа детали, сейчас нужно:

- менять SQL
- менять backend
- менять frontend

### 4.4 Плохая масштабируемость

Если добавить электродвигатели, гидроцилиндры, подшипники, ремни, датчики и т.д., таблица быстро превратится в набор случайных колонок, большинство из которых будут пустыми.

### 4.5 Поиск и отображение завязаны на старые поля

Сейчас поиск и выбор standard parts построен вокруг:

- `part_type`
- `designation`
- `standard_system`

Для новой модели понадобится отдельная стратегия отображаемого имени и индексируемых атрибутов.

## 5. Цель изменений

Нужно преобразовать каталог стандартных деталей в настраиваемый классификатор, где можно:

1. Создавать классы стандартных деталей.
2. Для каждого класса настраивать собственный набор полей.
3. Создавать конкретные позиции standard parts на основе выбранного класса.
4. Хранить значения полей без изменения схемы БД под каждый новый тип.
5. Сохранить `standard_part_id` как основную ссылочную сущность для OEM, supplier parts, RFQ и client requests.

## 6. Предлагаемая целевая модель

### 6.1 Общий принцип

Нужно разделить:

- класс стандартной детали
- определение полей класса
- конкретную стандартную деталь
- значения полей конкретной детали

### 6.2 Новые сущности

#### A. `standard_part_classes`

Справочник классов стандартных деталей.

Примеры:

- Болт
- Гайка
- Шайба
- Подшипник
- Электродвигатель
- Гидроцилиндр

Предлагаемые поля:

- `id`
- `code`
- `name`
- `description`
- `parent_id` nullable
- `is_active`
- `sort_order`
- `created_at`
- `updated_at`

`parent_id` нужен, если позже потребуется иерархия:

- Крепеж
- Крепеж / Болты
- Крепеж / Гайки

#### B. `standard_part_class_fields`

Описание полей, доступных для конкретного класса.

Предлагаемые поля:

- `id`
- `class_id`
- `code`
- `label`
- `field_type`
- `is_required`
- `is_in_list`
- `is_in_filters`
- `is_in_title`
- `is_searchable`
- `sort_order`
- `unit`
- `placeholder`
- `help_text`
- `default_value`
- `settings_json`
- `is_active`

`field_type`:

- `text`
- `textarea`
- `number`
- `boolean`
- `select`
- `multiselect`
- `date`

`settings_json` нужен для настроек:

- точность числа
- min/max
- regexp
- список допустимых значений
- режим автогенерации названия

#### C. `standard_part_field_options`

Опции для полей типа `select` и `multiselect`.

Предлагаемые поля:

- `id`
- `field_id`
- `value_code`
- `value_label`
- `sort_order`
- `is_active`

#### D. `standard_parts`

Главная сущность standard part должна сохраниться, но стать более универсальной.

Предлагаемая структура:

- `id`
- `class_id`
- `display_name`
- `display_name_norm`
- `designation` nullable
- `uom`
- `description_ru`
- `description_en`
- `notes`
- `attributes_search_text`
- `is_active`
- `created_at`
- `updated_at`

Комментарий:

- `standard_parts.id` сохраняется как основной внешний ключ всей системы
- `display_name` становится каноническим названием для таблиц, выборов и бизнес-процессов
- `designation` можно временно сохранить для обратной совместимости
- старые узкоспециальные колонки нужно убрать из бизнес-логики и затем удалить на финальном этапе

#### E. `standard_part_values`

Значения полей для конкретной standard part.

Предлагаемые поля:

- `id`
- `standard_part_id`
- `field_id`
- `value_text`
- `value_number`
- `value_boolean`
- `value_date`
- `value_json`
- `created_at`
- `updated_at`

Ожидаемое правило:

- в зависимости от `field_type` используется только один тип значения

## 7. Почему именно такая модель

### 7.1 Почему не делать отдельную таблицу под каждый тип

Это вернет нас к постоянным миграциям и усложнит связи.

### 7.2 Почему не удалять `standard_parts`

Потому что на `standard_parts.id` уже завязаны многие рабочие процессы. Это системная ссылка, ее нужно сохранить.

### 7.3 Почему нужен `display_name`

Сейчас интерфейсы выбора standard parts используют короткое человекочитаемое представление. После перехода на динамические поля нужен единый стабильный заголовок карточки.

### 7.4 Почему нужны настройки `is_in_title`, `is_in_list`, `is_in_filters`

Чтобы один и тот же классификатор управлял:

- формой создания
- карточкой просмотра
- колонками списка
- фильтрами
- составлением названия позиции

## 8. Функциональные требования

### 8.1 Управление классами standard parts

Система должна позволять:

- создавать класс standard part
- редактировать класс
- архивировать класс
- задавать родительский класс опционально

### 8.2 Управление полями класса

Для каждого класса должно быть возможно:

- добавить поле
- изменить поле
- отключить поле
- настроить обязательность
- настроить порядок
- настроить тип поля
- настроить отображение в списке
- настроить использование в фильтрах
- настроить участие в формировании названия

### 8.3 Управление конкретными standard parts

Пользователь должен иметь возможность:

- выбрать класс
- увидеть динамическую форму по настройкам класса
- создать standard part
- отредактировать значения полей
- деактивировать standard part
- искать standard parts по базовому тексту и по атрибутам

### 8.4 Отображение standard parts в других разделах

Во всех текущих связях должно остаться удобное отображение:

- класс standard part
- `display_name`
- основные атрибуты для быстрого чтения

Это нужно для:

- OEM карточек
- supplier parts
- client requests
- RFQ
- selection
- coverage

## 9. Нефункциональные требования

### 9.1 Обратная совместимость

На этапе миграции нужно сохранить работоспособность:

- существующих `standard_part_id`
- ссылок OEM -> standard
- ссылок supplier -> standard
- процессов client requests / RFQ / coverage

### 9.2 Производительность

Для быстрых списков и поиска нельзя собирать всю карточку только через тяжелые EAV-запросы.

Поэтому нужно хранить в `standard_parts`:

- `display_name`
- `attributes_search_text`
- возможно краткий `summary_json` на следующем этапе

### 9.3 Управляемая миграция

Рефакторинг нужно делать поэтапно, без одномоментного слома всех экранов.

## 10. Требования к API

Нужен новый слой API.

### 10.1 Классы

- `GET /standard-part-classes`
- `POST /standard-part-classes`
- `PUT /standard-part-classes/:id`
- `GET /standard-part-classes/:id`

### 10.2 Поля класса

- `GET /standard-part-classes/:id/fields`
- `POST /standard-part-classes/:id/fields`
- `PUT /standard-part-class-fields/:id`
- `POST /standard-part-class-fields/:id/options`

### 10.3 Standard parts

Нужен переход от фиксированного payload к универсальному.

Пример целевого payload:

```json
{
  "class_id": 12,
  "display_name": "Болт M8x16 ISO 4017 8.8 Zn",
  "uom": "pcs",
  "description_ru": "Болт шестигранный",
  "notes": null,
  "attributes": [
    { "field_code": "thread_diameter", "value": "M8" },
    { "field_code": "length_mm", "value": 16 },
    { "field_code": "standard", "value": "ISO 4017" },
    { "field_code": "strength_class", "value": "8.8" },
    { "field_code": "coating", "value": "Zn" }
  ]
}
```

### 10.4 Переходный период

На переходном этапе можно сохранить старый `/standard-parts`, но:

- вернуть в нем `class_id`
- вернуть `display_name`
- вернуть `attributes`
- старые поля отдавать как compatibility layer только временно

## 11. Требования к frontend

### 11.1 Новый раздел классификатора

Нужен интерфейс администрирования:

- список классов
- редактор полей класса
- управление опциями select-полей

### 11.2 Новая форма standard part

Форма должна строиться динамически из метаданных класса.

### 11.3 Список standard parts

Список должен поддерживать:

- фильтр по классу
- текстовый поиск
- динамические фильтры по полям класса
- настраиваемые колонки по настройкам поля

### 11.4 Использование в OEM и supplier parts

Пикеры выбора standard parts должны работать не по старым колонкам, а по:

- имени класса
- `display_name`
- краткому описанию

## 12. Миграция данных

Так как данных сейчас почти нет, миграцию можно делать прагматично.

### 12.1 Шаги миграции

1. Создать новые таблицы классов, полей, опций и значений.
2. Добавить в `standard_parts` колонку `class_id`.
3. Добавить в `standard_parts` колонку `display_name`.
4. Добавить в `standard_parts` колонку `attributes_search_text`.
5. Создать базовый класс `Болт`.
6. Создать поля класса `Болт`.
7. Перенести значения существующих колонок текущих записей в `standard_part_values`.
8. Сформировать `display_name` для старых записей.
9. Перевести backend на новую модель чтения и записи.
10. Перевести frontend на динамический интерфейс.
11. Только после стабилизации удалить старые специализированные колонки из `standard_parts`.

### 12.2 Принцип удаления старых колонок

Удалять колонки:

- `strength_class`
- `material_spec`
- `coating`
- `thread_spec`
- `size_note`
- возможно `standard_system`
- возможно `part_type`

нужно только после полной адаптации backend и frontend.

На раннем этапе они могут оставаться как compatibility слой.

## 13. Ограничения и риски

### 13.1 Главный риск

Сломать существующие процессы, где standard part уже участвует как нормализованная ссылка.

### 13.2 Риск по UX

Если сделать чистый EAV без `display_name` и без правил отображения, каталоги и пикеры станут неудобными.

### 13.3 Риск по фильтрации

Динамические поля потребуют отдельной логики индексации и построения SQL.

### 13.4 Риск по сложности

Если сразу пытаться сделать "универсальный конструктор всего", проект можно перегрузить. Нужен практичный первый релиз.

## 14. Рекомендуемый scope первой реализации

Для первой версии рекомендую сделать не весь "суперконструктор", а рабочий минимум:

### V1

- классы standard parts
- поля классов
- значения полей
- динамическая форма создания и редактирования
- `display_name`
- фильтр по классу
- текстовый поиск
- обновленные пикеры OEM/supplier

### Не включать в V1

- глубокую иерархию классов
- сложные вычисляемые поля
- автоматический конструктор любых табличных представлений
- сложные правила зависимых полей

## 15. Предлагаемый план реализации после согласования

### Этап 1. SQL

- новые таблицы классификатора
- расширение `standard_parts`
- начальная миграция текущих данных

### Этап 2. Backend

- новые CRUD routes для классов и полей
- рефакторинг `/standard-parts`
- compatibility layer на переходный период

### Этап 3. Frontend

- новый экран управления классификатором
- новая динамическая форма standard parts
- обновление OEM/supplier picker'ов

### Этап 4. Стабилизация

- проверка связей в RFQ / client requests / supplier flows
- отказ от старых жестких полей

## 16. Решения, которые предлагается утвердить

Нужно утвердить следующие принципы:

1. `standard_parts` остается основной сущностью и сохраняет свои `id`.
2. Вводится новый слой `class -> fields -> values`.
3. Карточка standard part становится динамической.
4. Старые фиксированные поля переводятся в compatibility mode и потом удаляются.
5. Первая версия делается как прагматичный V1 без избыточной универсальности.

## 17. Краткий вывод по анализу

Текущая реализация действительно структурно неверна для развития каталога стандартных деталей.

Проблема не в конкретных колонках, а в том, что сейчас отсутствует отдельная сущность "класс стандартной детали" и отсутствует механизм настройки атрибутов класса.

Правильное направление:

- не заменять `standard_parts` другой сущностью
- а превратить текущий каталог в классификатор с динамическими атрибутами
- сохранив `standard_part_id` как основной идентификатор во всей системе

## 18. Что делать после утверждения

После утверждения этого ТЗ следующий рабочий шаг:

1. подготовить SQL-проект новой схемы
2. описать переходную модель API
3. реализовать backend
4. затем реализовать frontend

- применяемость по моделям `oem_part_model_fitments`
- BOM `oem_part_model_bom`
- альтернативы OEM `oem_part_alt_groups` + `oem_part_alt_items`
- связи с supplier parts `supplier_part_oem_parts`
- комплекты поставщиков `supplier_bundles` + `supplier_bundle_items` + `supplier_bundle_item_links`
- presentation profile / отображаемые номера
- client request / RFQ / selection

### 19.2 Реальный поток бизнес-процесса сейчас

#### Шаг 1. Каталог OEM

Пользователь создает или ведет OEM-деталь:

- производитель
- OEM каталожный номер
- описание
- применяемость по моделям

Это базовая инженерная сущность каталога.

#### Шаг 2. OEM-деталь получает структуру

На уровне OEM-детали живут:

- BOM
- альтернативные OEM
- материалы
- документы
- presentation profile
- связи с поставщиками
- комплекты поставщика
- вкладка standard parts

То есть OEM-деталь сейчас является “корнем карточки”.

#### Шаг 3. Supplier part связывается с OEM

Сейчас основная закупочная логика аналогов и OEM-предложений идет через:

- `supplier_part_oem_parts`

Именно эта таблица сейчас используется как главный мост:

- от OEM-позиции к предложениям поставщиков
- от supplier hints в RFQ к candidate parts

#### Шаг 4. Client request и RFQ

В client request и RFQ основной путь такой:

1. Клиент присылает OEM номер или запрос на OEM позицию.
2. В `client_request_revision_items` сохраняется `oem_part_id`.
3. Из этого рождается `rfq_item`.
4. Для `rfq_item` строятся компоненты:
- либо сама OEM деталь
- либо BOM-компоненты
- либо комплектные роли
5. Для supplier hints система ищет supplier parts в первую очередь по `supplier_part_oem_parts`.

#### Шаг 5. Selection и response lines

Когда поставщик отвечает:

- строки ответа и selection тоже в первую очередь разрешаются через `oem_part_id`
- `standard_part_id` в схеме присутствует, но не является основным маршрутом подбора

### 19.3 Где именно standard parts сейчас встроены слабо

`standard_part_id` уже добавлен в схему многих таблиц, но фактически почти не используется в рабочих данных.

Проверка БД на 2026-03-23:

- `client_request_revision_items`: `57 / 0` со `standard_part_id`
- `client_request_revision_item_components`: `57 / 0`
- `rfq_items`: `57 / 0`
- `rfq_response_lines.standard_part_id`: `112 / 0`
- `rfq_response_lines.requested_standard_part_id`: `112 / 0`
- `rfq_supplier_line_selections`: `115 / 0`
- `rfq_coverage_option_lines`: `97 / 0`

Фактически это означает:

- standard part присутствует в архитектуре как подготовленная ссылка
- но бизнес-поток еще не переключен на нее

### 19.4 Как сейчас работают supplier hints

В генерации supplier hints и RFQ Excel / selection-строк основная логика ищет подсказки:

- по `oem_part_id`
- по `bundle_item_id`

Это хорошо работает для:

- OEM-позиций
- BOM-компонентов, если они тоже OEM
- комплектов поставщика

Но это плохо работает для стандартных изделий, потому что:

- поставщик болтов обычно продает не “OEM номер Metso” и не “OEM номер Sandvik”
- он продает каноническое изделие типа `Bolt M8x16`

Именно здесь у текущей архитектуры и появляется концептуальный разрыв.

## 20. Что была за правильная идея standard parts

Изначальная бизнес-идея standard parts по смыслу правильная:

- один и тот же физический стандартный элемент может иметь разные OEM номера у разных производителей оборудования
- закупаться он должен не как “OEM номер производителя”, а как каноническое стандартное изделие

Пример:

- стандартная деталь: `Болт M8x16`
- у производителя A: OEM номер `A-12345`
- у производителя B: OEM номер `B-98765`
- у производителя C: OEM номер `C-55555`

Все три OEM позиции по сути должны указывать на одну каноническую standard part.

Тогда поставщик болтов должен связываться:

- не с тремя разными OEM-номерами по отдельности
- а с одной канонической standard part

## 21. Ключевое архитектурное различие, которое нужно закрепить

Нужно различать две сущности:

### 21.1 Standard part

Это каноническая инженерно-закупочная сущность.

Примеры:

- болт M8x16 ISO 4017 8.8 Zn
- гайка M8 ISO 4032 8
- подшипник 6205-2RS
- электродвигатель 7.5kW 1500rpm B3 380V

Это то, что:

- имеет реальные характеристики
- закупается у универсальных поставщиков
- может повторяться у разных OEM-каталогов

### 21.2 OEM representation of standard part

Это представление стандартной детали внутри конкретного OEM-каталога.

То есть:

- конкретный OEM номер
- конкретный производитель оборудования
- конкретная модель или группа моделей
- привязка к BOM и к OEM-структуре

Именно эта сущность должна попадать в OEM-каталог и в BOM.

## 22. Вывод по вариантам реализации

### 22.1 Вариант A. Оставить только вкладку standard parts внутри OEM детали

Плюсы:

- минимальные изменения
- логика уже частично есть

Минусы:

- standard part остается вторичной сущностью
- пользователь мыслит от OEM-карточки, а не от канонического изделия
- тяжело массово связать одну standard part с несколькими OEM номерами
- неудобно строить закупочную логику “поставщик болтов”

Вывод:

- как основной подход не рекомендуется

### 22.2 Вариант B. Делать назначение OEM <-> standard из таблицы standard parts

Плюсы:

- standard part становится первичной сущностью
- удобно видеть все OEM-представления одного изделия
- удобно видеть всех supplier parts одного стандартного изделия
- удобно развивать закупочный сценарий

Минусы:

- нужен новый UX для массовой привязки и создания OEM-представлений

Вывод:

- это должен быть основной центр работы со standard parts

### 22.3 Вариант C. Создавать “OEM standard” из карточки standard part

Это наиболее сильный вариант.

Сценарий:

1. В каталоге standard parts создается каноническая деталь.
2. Пользователь выбирает:
- производителя оборудования
- модель или модели
- OEM номер
- описание OEM-представления
3. Система автоматически создает новую `oem_part`, уже связанную с этой standard part.
4. Эта новая OEM-деталь дальше участвует:
- в BOM
- в fitments
- в client request
- в RFQ
- в selection

Плюсы:

- сохраняется OEM-центричная операционная модель системы
- при этом канонический источник истины для стандартных деталей становится правильным
- пользователь работает не через “сначала OEM, потом привязать standard”, а через “сначала каноническая standard part, потом породить OEM-представления”

Вывод:

- это рекомендуемый целевой подход

## 23. Рекомендуемая целевая архитектура

### 23.1 Что должно остаться OEM-центричным

На `oem_parts` должны остаться:

- BOM
- применяемость по моделям
- альтернативные OEM
- presentation profile
- документы
- материалы
- комплекты поставщика
- клиентские и RFQ процессы

### 23.2 Что должно стать standard-центричным

На `standard_parts` должны перейти:

- каноническая инженерная карточка стандартной детали
- динамические атрибуты
- связи с universal suppliers
- канонические supplier offers для standard items
- массовое создание OEM-представлений для производителей/моделей

### 23.3 Новая промежуточная сущность

Нужно рассмотреть не просто связь `oem_part_standard_parts`, а более сильную семантику:

- один `oem_part` для стандартной OEM-позиции должен указывать на одну основную `standard_part`

По сути текущее `oem_part_standard_parts` лучше эволюционировать в модель:

- основная standard part для OEM-позиции
- возможно дополнительные secondary links только при необходимости

Практически это значит:

- для стандартной OEM-позиции должен быть один основной canonical standard reference
- многозначные связи допустимы только как исключение

## 24. Рекомендация по UX

### 24.1 Где НЕ делать основной сценарий

Не делать главным сценарием экран:

- открыть OEM деталь
- перейти на вкладку “Стандартные детали”
- вручную добавить standard part

Этот сценарий можно оставить как вспомогательный, но не как основной.

### 24.2 Где делать основной сценарий

Основной сценарий должен быть в каталоге standard parts:

1. Создать каноническую standard part.
2. Настроить ее атрибуты.
3. Посмотреть:
- связанные OEM-представления
- связанные supplier parts
4. Нажать действие:
- “Создать OEM-представление”
- или “Привязать к существующей OEM-детали”

### 24.3 Что должна делать команда “Создать OEM-представление”

Команда должна позволять:

- выбрать производителя оборудования
- выбрать модель или несколько моделей
- ввести OEM номер
- ввести OEM-описание
- при необходимости сразу открыть созданную OEM-карточку

После этого система должна:

1. создать `oem_parts`
2. создать fitments для выбранных моделей
3. создать связь с `standard_part`
4. сделать эту связь основной

## 25. Что это даст в бизнес-процессе

### 25.1 Для клиента

Клиент по-прежнему может прислать OEM номер конкретного производителя.

### 25.2 Для инженерного каталога

В каталоге этот OEM номер будет лежать как отдельная `oem_part`, но связанная с канонической standard part.

### 25.3 Для закупки

Система сможет понимать:

- это не уникальная OEM-позиция
- это каноническая standard part
- искать надо не только поставщиков по OEM номеру, но и поставщиков standard item

### 25.4 Для RFQ

Тогда в RFQ можно будет добавлять отдельную логику:

- если у компонента есть `standard_part_id`
- система ищет кандидатов не только в `supplier_part_oem_parts`
- но и в `supplier_part_standard_parts`

Это и есть путь к сценарию:

- “вот есть поставщик, который делает болты”

## 26. Обновленное решение для реализации

Рекомендуется утвердить следующее:

1. `standard_parts` становится каноническим каталогом стандартных изделий.
2. `oem_parts` остается операционным центром BOM/RFQ/client flows.
3. Для стандартных OEM-номеров пользователь должен иметь возможность создавать `oem_part` прямо из карточки standard part.
4. Основной UX управления привязками standard <-> OEM переносится в каталог standard parts.
5. Вкладка standard parts внутри OEM-карточки остается, но как вторичный/вспомогательный способ редактирования.
6. После этого supplier hints и RFQ должны научиться искать кандидатов и по `standard_part_id`, а не только по `oem_part_id`.

## 27. Утвержденная структура интерфейса

### 27.1 Отдельный раздел

Принято решение:

- не расширять текущий `Классификатор оборудования`
- сделать отдельный раздел `Классификатор стандартных деталей`

Причина:

- классификатор оборудования описывает технику и типы машин
- классификатор standard parts описывает типы стандартных изделий и их атрибуты
- это разные предметные области, и смешивать их в одну иерархию нельзя

### 27.2 Общий UX-принцип

Нужно использовать похожую механику интерфейса, как в `Классификаторе оборудования`, но с другой сущностью.

То есть:

- слева дерево или список классов standard parts
- справа workspace выбранного класса

### 27.3 Не две отдельные страницы, а один workspace

Принято решение:

- не делать две независимые страницы “настройка сущностей” и “наполнение”
- делать один основной экран-workspace

Причина:

- пользователь мыслит от выбранного класса
- в одном месте должно быть видно:
  - структура класса
  - стандартные детали этого класса
  - OEM-представления
  - связанные поставщики

## 28. Предлагаемая структура экранов

### 28.1 Раздел 1. `Классификатор стандартных деталей`

Основной инженерный workspace.

Левая часть:

- дерево классов standard parts

Примеры узлов:

- Крепеж
- Крепеж / Болты
- Крепеж / Гайки
- Подшипники
- Электрооборудование
- Электрооборудование / Электродвигатели
- Гидравлика
- Гидравлика / Гидроцилиндры

Правая часть:

- workspace выбранного класса

### 28.2 Вкладки workspace класса

Для выбранного класса рекомендованы вкладки:

#### Вкладка `Структура класса`

Назначение:

- создать и редактировать поля класса
- управлять обязательностью
- управлять типами полей
- управлять отображением в списке, фильтрах и заголовке

#### Вкладка `Стандартные детали`

Назначение:

- список канонических standard parts выбранного класса
- создание новой standard part
- редактирование значений
- фильтрация по атрибутам класса

#### Вкладка `OEM-представления`

Назначение:

- видеть все `oem_part`, связанные со standard parts этого класса
- создавать OEM-представления из канонической standard part
- привязывать existing OEM к standard part

#### Вкладка `Поставщики`

Назначение:

- видеть supplier parts, привязанные к standard parts этого класса
- понимать, какие поставщики покрывают данный класс изделий

### 28.3 Раздел 2. `Стандартные детали`

Отдельную текущую страницу `Стандартные детали` имеет смысл сохранить, но изменить ее роль.

Она должна стать:

- плоским глобальным каталогом всех standard parts
- быстрым поиском по всему реестру
- удобной операционной таблицей

Она не должна быть основным местом настройки структуры классов.

## 29. Как будет работать интерфейс для пользователя

### 29.1 Создание нового класса standard parts

Сценарий:

1. Пользователь открывает раздел `Классификатор стандартных деталей`.
2. В левой панели нажимает `Создать класс`.
3. Вводит:
- название класса
- код класса
- родительский класс опционально
4. Сохраняет класс.
5. В правой части открывается workspace нового класса.

### 29.2 Настройка полей класса

Пользователь открывает вкладку `Структура класса`.

Для каждого поля он может задать:

- название
- код поля
- тип поля
- обязательность
- единицу измерения
- порядок
- участвует ли поле в названии
- показывать ли в таблице
- использовать ли в фильтрах
- делать ли поле searchable

Если поле типа `select`, пользователь добавляет список значений.

### 29.3 Создание standard part

Пользователь открывает вкладку `Стандартные детали`.

Сценарий:

1. Нажимает `Создать standard part`.
2. Система строит форму автоматически по полям выбранного класса.
3. Пользователь заполняет значения.
4. Система:
- сохраняет запись в `standard_parts`
- сохраняет значения в `standard_part_values`
- формирует `display_name`
- формирует `attributes_search_text`

### 29.4 Создание OEM-представления из standard part

Пользователь открывает карточку standard part.

Далее:

1. Нажимает `Создать OEM-представление`.
2. Выбирает:
- производителя оборудования
- модель или несколько моделей
3. Вводит:
- OEM номер
- OEM описание
- дополнительные примечания при необходимости
4. Сохраняет.

Система автоматически:

1. создает `oem_part`
2. создает `oem_part_model_fitments`
3. создает основную связь `oem_part -> standard_part`
4. возвращает пользователя либо в карточку standard part, либо открывает новую OEM-карточку

### 29.5 Привязка existing OEM к standard part

Дополнительно должен быть сценарий:

1. открыть standard part
2. нажать `Привязать existing OEM`
3. найти OEM деталь по:
- производителю
- модели
- OEM номеру
4. выбрать запись
5. создать основную canonical link

- `Электродвигатель 7.5kW 380V 1500rpm B3 IP55`

### 30.4 Создание OEM-представления

Из карточки этой standard part пользователь нажимает:

- `Создать OEM-представление`

Далее выбирает:

- производитель оборудования: `Metso`
- модель: `HP800`

И вводит:

- OEM номер: `MM-ELM-7500-01`
- OEM описание: `Главный электродвигатель привода`

Система создает:

- новую `oem_part`
- fitment на модель `HP800`
- основную связь с канонической standard part

После этого:

- в BOM можно использовать уже OEM номер `MM-ELM-7500-01`
- при этом система знает, что канонически это `Электродвигатель 7.5kW 380V 1500rpm B3 IP55`

### 30.5 Закупочный эффект

Позже supplier parts могут привязываться напрямую к этой standard part:

- supplier A продает этот мотор как generic item
- supplier B продает OEM-аналог
- supplier C продает мотор под своим внутренним номером

Тогда RFQ в будущем сможет искать:

- по OEM номеру
- по canonical standard part

## 31. Технический принцип хранения данных

Важно:

- интерфейс НЕ создает новые физические таблицы в БД под каждый класс

Это не должно работать так:

- пользователь создал класс `Электродвигатели`
- система создала новую таблицу `standard_parts_electric_motors`

Так делать нельзя.

### 31.1 Правильная модель хранения

Нужно использовать мета-модель:

- `standard_part_classes`
- `standard_part_class_fields`
- `standard_part_field_options`
- `standard_parts`
- `standard_part_values`

То есть UI создает:

- описание класса
- описание полей класса
- значения полей

Но не создает SQL-таблицы.

### 31.2 Что это означает practically

Если пользователь создал класс `Электродвигатели` и поле `power_kw`, то в БД появятся:

- строка в `standard_part_classes`
- строка в `standard_part_class_fields`

А когда пользователь создаст конкретный мотор:

- строка в `standard_parts`
- несколько строк в `standard_part_values`

### 31.3 Почему это правильно

Потому что такой подход:

- не требует миграций под каждый новый класс
- позволяет строить универсальный backend
- позволяет строить динамический frontend
- сохраняет управляемость данных

## 32. Финально утвержденный UX-подход

Утверждено:

1. Отдельный раздел `Классификатор стандартных деталей`.
2. Отдельный плоский каталог `Стандартные детали` остается как операционный список.
3. Основной экран классификатора делается как один workspace, а не две независимые страницы.
4. Внутри workspace пользователь и настраивает класс, и наполняет его данными.
5. Основной сценарий создания OEM standard идет из карточки standard part.
6. Текущая вкладка standard parts внутри OEM-карточки остается только как вспомогательный способ редактирования или может быть удалена на финальном этапе, если новый сценарий полностью покрывает работу.

## 33. Предреализационный технический блок

### 33.1 Базовое решение по внедрению

Принято следующее:

- старые записи `standard_parts` можно удалить
- старые связи `oem_part_standard_parts` можно удалить
- обратную совместимость со старой логикой standard parts специально сохранять не нужно
- но нужно сохранить саму сущность `standard_parts` как основной идентификатор новой канонической standard part

Это значит:

- делаем чистую переделку standard parts
- без сложной data migration старого контента
- с минимально необходимой адаптацией действующих бизнес-процессов

## 34. Целевая схема БД

### 34.1 Таблица `standard_part_classes`

Назначение:

- дерево или список классов стандартных изделий

Предлагаемая структура:

- `id` INT PK AI
- `parent_id` INT NULL FK -> `standard_part_classes.id`
- `code` VARCHAR(100) NOT NULL UNIQUE
- `name` VARCHAR(255) NOT NULL
- `description` TEXT NULL
- `sort_order` INT NOT NULL DEFAULT 0
- `is_active` TINYINT(1) NOT NULL DEFAULT 1
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

Индексы:

- `idx_spc_parent_id`
- `idx_spc_code`
- `idx_spc_active`

### 34.2 Таблица `standard_part_class_fields`

Назначение:

- описание полей класса

Предлагаемая структура:

- `id` INT PK AI
- `class_id` INT NOT NULL FK -> `standard_part_classes.id`
- `code` VARCHAR(100) NOT NULL
- `label` VARCHAR(255) NOT NULL
- `field_type` ENUM('text','textarea','number','boolean','select','multiselect','date') NOT NULL
- `sort_order` INT NOT NULL DEFAULT 0
- `is_required` TINYINT(1) NOT NULL DEFAULT 0
- `is_active` TINYINT(1) NOT NULL DEFAULT 1
- `is_in_title` TINYINT(1) NOT NULL DEFAULT 0
- `is_in_list` TINYINT(1) NOT NULL DEFAULT 0
- `is_in_filters` TINYINT(1) NOT NULL DEFAULT 0
- `is_searchable` TINYINT(1) NOT NULL DEFAULT 0
- `unit` VARCHAR(50) NULL
- `placeholder` VARCHAR(255) NULL
- `help_text` TEXT NULL
- `default_value` VARCHAR(255) NULL
- `settings_json` JSON NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

Ограничения:

- UNIQUE (`class_id`, `code`)

Индексы:

- `idx_spcf_class_id`
- `idx_spcf_class_sort`
- `idx_spcf_type`

### 34.3 Таблица `standard_part_field_options`

Назначение:

- варианты значений для `select` и `multiselect`

Предлагаемая структура:

- `id` INT PK AI
- `field_id` INT NOT NULL FK -> `standard_part_class_fields.id`
- `value_code` VARCHAR(100) NOT NULL
- `value_label` VARCHAR(255) NOT NULL
- `sort_order` INT NOT NULL DEFAULT 0
- `is_active` TINYINT(1) NOT NULL DEFAULT 1

Ограничения:

- UNIQUE (`field_id`, `value_code`)

Индексы:

- `idx_spfo_field_id`

### 34.4 Таблица `standard_parts`

Назначение:

- канонические standard parts

Текущую таблицу `standard_parts` нужно очистить и привести к новой структуре.

Предлагаемая структура:

- `id` INT PK AI
- `class_id` INT NOT NULL FK -> `standard_part_classes.id`
- `display_name` VARCHAR(500) NOT NULL
- `display_name_norm` VARCHAR(500) GENERATED / либо обычная колонка
- `designation` VARCHAR(255) NULL
- `uom` VARCHAR(16) NOT NULL DEFAULT 'pcs'
- `description_ru` TEXT NULL
- `description_en` TEXT NULL
- `notes` TEXT NULL
- `attributes_search_text` TEXT NULL
- `is_active` TINYINT(1) NOT NULL DEFAULT 1
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

Индексы:

- `idx_sp_class_id`
- `idx_sp_active`
- `idx_sp_display_name`
- полнотекстовый или альтернативный индекс по `attributes_search_text` в зависимости от выбранной стратегии

### 34.5 Таблица `standard_part_values`

Назначение:

- значения полей standard part

Предлагаемая структура:

- `id` INT PK AI
- `standard_part_id` INT NOT NULL FK -> `standard_parts.id`
- `field_id` INT NOT NULL FK -> `standard_part_class_fields.id`

Ограничения:

Индексы:

- `idx_spv_standard_part_id`

- `idx_spv_value_number`
- `idx_spv_value_boolean`
- `idx_spv_value_date`

### 34.6 Таблица связи OEM -> canonical standard part

Текущую таблицу `oem_part_standard_parts` можно сохранить по имени, но упростить по смыслу.

Рекомендуемое направление:

- одна основная canonical link на OEM-позицию

Минимальная структура:

- `oem_part_id` INT NOT NULL FK -> `oem_parts.id`
- `standard_part_id` INT NOT NULL FK -> `standard_parts.id`
- `is_primary` TINYINT(1) NOT NULL DEFAULT 1
- `note` VARCHAR(255) NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

Правило:

- для standard OEM-позиций должна существовать одна primary canonical link
- дополнительные links допускаются только если реально нужны

### 34.7 Таблица связи supplier part -> canonical standard part

Текущую таблицу `supplier_part_standard_parts` сохранить.

Назначение:

- supplier part может поставлять canonical standard item напрямую

- `supplier_part_id`

## 38. Как меняется логика RFQ и supplier hints

### 38.1 Текущее состояние

Сейчас supplier hints ищут кандидатов почти только по:

- `oem_part_id`
- `bundle_item_id`

### 38.2 Целевое состояние

Если у позиции или компонента есть `standard_part_id`, система должна:

1. сначала искать supplier parts по `supplier_part_standard_parts`
2. затем дополнительно искать supplier parts по `supplier_part_oem_parts`, если есть OEM-представление
3. агрегировать кандидатов
4. показывать, откуда найден match:
- `STANDARD`
- `OEM`
- `BOTH`

### 38.3 Практическое правило

Для canonical standard item supplier link по standard part должен стать первичным сигналом закупки.
