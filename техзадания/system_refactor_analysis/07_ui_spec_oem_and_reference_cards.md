# Прикладной UI-spec: новая OEM карточка и карточка reference entity

## 1. Зачем нужен этот файл

До этого спецификации были в основном про архитектуру, таблицы и миграции.

Этот файл прикладной:

- как должен выглядеть новый пользовательский путь;
- что именно должно быть в карточке OEM;
- что именно должно быть в карточке unified reference entity;
- что переносится, что остается, что становится legacy.

Это нужно, чтобы не запутаться во время переделки фронта.

---

## 2. Главный UX-принцип

Пользователь не должен перестать узнавать систему.

Значит:

- OEM карточка остается;
- OEM список остается;
- но в них появляется новый центр: `каноническая сущность`.

Новый путь пользователя:

1. он может войти как раньше через OEM;
2. он видит, что у OEM есть каноническая сущность;
3. он может открыть карточку канонической сущности;
4. постепенно начинает работать уже не только через OEM, но и через unified reference.

То есть UI эволюционный, а не революционный.

---

## 3. Новая карточка OEM детали

## 3.1. Общая идея

OEM карточка больше не должна пытаться быть единственным центром всего.

Она должна отвечать на вопрос:

- что это за OEM-представление;
- где оно применяется;
- как оно участвует в BOM;
- как оно связано с канонической сущностью;
- как его закупать.

### Новая структура карточки

Я предлагаю делить OEM карточку на 4 верхних блока:

1. `OEM Identity`
2. `Canonical`
3. `Engineering`
4. `Procurement`

---

## 3.2. Верхняя шапка карточки OEM

### Что должно быть в header

- `OEM number`
- `manufacturer`
- `primary model`
- `RU description`
- short status badge

### Новые статусные бейджи

- `Канонизирована`
- `Требует уточнения`
- `Без канонической сущности`
- `Стандартная`
- `Уникальная`

### Что убрать из шапки

Не надо перегружать header:

- материалами;
- supplier info;
- document count;
- machine overrides.

Это должно жить ниже.

---

## 3.3. Блок `OEM Identity`

Этот блок должен заменить разрозненное базовое представление OEM детали.

### Поля блока

- производитель
- OEM номер
- описание RU
- описание EN
- тех. описание
- ТН ВЭД
- UOM
- `has_drawing`
- `is_overweight`
- `is_oversize`
- fitment summary

### Дополнительно

Отдельной строкой:

- `legacy OEM group`

Но:

- это поле должно быть помечено как `legacy`;
- редактирование надо постепенно убирать;
- оно не должно восприниматься как основной классификатор системы.

### Действия блока

- редактировать OEM identity
- удалить из модели
- удалить полностью

---

## 3.4. Блок `Canonical`

Это самый важный новый блок.

Он должен заменить старую вкладку `Стандартные детали`.

### Что показываем

- linked `reference entity`
- `entity type`
- `canonical status`
- primary taxonomy path
- краткий canonical title

### Если связь есть

Показываем:

- название канонической сущности
- тип сущности
- статус:
  - `draft`
  - `partially classified`
  - `canonicalized`
  - `supplier ready`
- кнопки:
  - `Открыть карточку сущности`
  - `Сменить связь`
  - `Отвязать`

### Если связи нет

Показываем:

- сообщение `Каноническая сущность не назначена`
- кнопки:
  - `Создать сущность`
  - `Привязать к существующей`

### Если сущность draft/неуточненная

Показываем warning:

- `Эта OEM позиция еще не канонизирована полностью`

И quick actions:

- выбрать тип сущности
- заполнить ключевые поля
- назначить таксономию

### Дополнительный блок внутри canonical

`Представления этой сущности`

Краткая summary:

- OEM representations count
- supplier representations count
- internal codes count

Это поможет пользователю понять, что OEM деталь теперь одна из репрезентаций, а не весь мир.

---

## 3.5. Блок `Engineering`

Сюда надо собрать все инженерное и эксплуатационное.

### Вкладки блока

1. `BOM`
2. `Где используется`
3. `Материалы`
4. `По машинам клиентов`
5. `Документы`
6. `Альтернативы`

### 1. `BOM`

Оставить почти как сейчас:

- дерево
- редактирование количества
- bulk add
- переход в дочернюю OEM карточку

Но добавить сверху маленькую canonical-подсказку:

- если у дочерней детали есть canonical entity, показывать ее значок/тип.

### 2. `Где используется`

Оставить как есть.

Дополнение:

- рядом с parent OEM part показывать canonical entity, если есть.

### 3. `Материалы`

Оставить по логике как есть:

- список материалов
- default material
- specs per material

Но переименовать смыслово:

- не просто `Материалы`, а `Материалы OEM-представления`

Это важно, потому что позже у канонической сущности тоже может быть material knowledge.

### 4. `По машинам клиентов`

Оставить как есть.

Это отдельный сильный слой.

Но визуально лучше показать 2 уровня:

- `Базовая применяемость по модели`
- `Отклонения по конкретным машинам`

### 5. `Документы`

Оставить как есть.

Рекомендация:

- потом ввести тип документа:
  - чертеж
  - фото
  - PDF
  - спецификация
  - сертификат

### 6. `Альтернативы`

Оставить.

Но переименовать:

- `Альтернативные OEM`

Чтобы было ясно, что это альтернативы внутри OEM-слоя, а не вообще все аналоги.

---

## 3.6. Блок `Procurement`

Сюда собрать закупочную часть.

### Вкладки блока

1. `Поставщики`
2. `Комплекты поставщика`
3. `OEM представление для RFQ`

### 1. `Поставщики`

Оставить текущие direct links.

Но добавить секции:

- `Exact supplier links`
- `Рекомендуемые поставщики`

В рекомендуемых показывать:

- кто подходит по exact canonical match;
- кто подходит по категории;
- кто подходит по специализации.

### 2. `Комплекты поставщика`

Оставить как есть.

Это полезный procurement abstraction.

### 3. `OEM представление для RFQ`

Это бывшая вкладка `Номера и видимость`.

Ее лучше переименовать, чтобы смысл был прозрачным.

Показывать:

- внутренний номер
- внутреннее имя
- supplier-visible номер
- supplier-visible description
- drawing code
- use by default in supplier RFQ

Почему так лучше:

- становится ясно, что это не про каноническую сущность;
- это именно процессное представление OEM в закупке.

---

## 4. Что убрать из OEM карточки как самостоятельные центры

### Убрать

- отдельную идею `Стандартные детали` как самостоятельный мир;
- `OEM groups` как важную классификацию.

### Заменить

- `Стандартные детали` -> `Каноническая сущность`
- `Группа` -> `Legacy OEM group` + taxonomy summary

---

## 5. Новая карточка `Reference Entity`

Это второй важный UI-центр системы.

Если OEM карточка отвечает на вопрос:

- `что это у производителя`

То карточка reference entity отвечает:

- `что это вообще такое`

---

## 5.1. Header карточки reference entity

### Что показываем

- `canonical_name`
- `entity type`
- `status`
- primary taxonomy path
- quick badges

### Бейджи

- `Standard item`
- `Unique part`
- `Assembly`
- `Equipment model`
- `Supplier ready`
- `Draft`

### Быстрые действия

- редактировать поля
- открыть OEM representations
- открыть поставщиков
- открыть связи

---

## 5.2. Вкладки карточки reference entity

Я бы делал такие вкладки.

### 1. `Основное`

Показывать:

- canonical name
- designation
- descriptions
- status
- notes
- type summary

### 2. `Поля`

Это динамическая вкладка.

Здесь рендерятся все поля типа сущности:

- числовые
- текстовые
- boolean
- select
- date

Для пользователя это будет главный интерфейс настройки сущности.

### 3. `Таксономии`

Показывать:

- дерево классификации
- текущие taxonomy links
- primary taxonomy

### 4. `Связи`

Показывать relation graph в виде списков:

- part_of
- contains
- used_in
- alternative_to
- substitute_for
- compatible_with

### 5. `OEM представления`

Показывать:

- все `oem_parts`, связанные с этой сущностью
- производитель
- OEM номер
- fitment summary
- признак primary

Отсюда же можно:

- создавать новую OEM representation;
- связывать существующую OEM деталь.

### 6. `Поставщики`

Показывать:

- supplier parts linked to this entity
- exact matches
- preferred suppliers
- supplier capabilities

### 7. `Материалы`

Это уже не OEM-specific, а canonical material knowledge.

Показывать:

- базовые рекомендованные материалы;
- material families;
- notes;
- optional reference specs.

Важно:

- не путать с `Материалы OEM-представления`.

### 8. `Применяемость`

Показывать:

- в каких equipment models встречается сущность;
- в каких узлах;
- summary по OEM fitments.

---

## 6. Как пользователь должен двигаться между карточками

### Сценарий 1. От OEM к reference

1. Пользователь открыл OEM деталь
2. Видит canonical block
3. Открывает reference entity
4. Видит все OEM/supplier/taxonomy связи

### Сценарий 2. От reference к OEM

1. Пользователь нашел сущность через taxonomy/filter
2. Открыл reference entity
3. Смотрит OEM representations
4. Уходит в конкретную OEM карточку

### Сценарий 3. Закупщик

1. Открыл OEM деталь из заказа
2. Видит canonical entity
3. Смотрит recommended suppliers
4. При необходимости идет в supplier parts / bundle

Если не делать сразу идеальный новый фронт, то MVP такой:

### Шаг 1

В OEM карточку добавить canonical block.

### Шаг 2

Сделать простую reference entity card:

- `Основное`
- `Поля`
- `OEM представления`

### Шаг 3

Потом добавлять:

- `Таксономии`
- `Связи`
- `Поставщики`

Это даст максимальную пользу без большого перегруза команды.

---

## 9. Что даст такой UI

Пользователь перестанет путаться между тремя разными центрами:

- OEM
- standard parts
- equipment classifier

Вместо этого у него будет понятная модель:

- OEM карточка: производитель и каталожный номер
- Reference entity: сущность как объект знания
- Taxonomy: классификация
- Suppliers: кто может это поставить

То есть интерфейс начнет отражать правильную доменную модель, а не исторически накопившиеся таблицы.
