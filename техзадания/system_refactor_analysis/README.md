# Анализ текущей OEM/каталожной архитектуры

Этот набор файлов нужен как рабочая карта для пересборки системы без потери логики.

Состав:

- `01_oem_front_to_db_map.md`
  Экран OEM-каталога и карточки детали: что видно на фронтенде, какие вкладки есть, какие API они дергают, какие таблицы стоят за этим.

- `02_oem_tables_and_domain_logic.md`
  Подробное описание OEM-таблиц, их ролей, связей и того, что в них уже является зачатком будущего единого справочника.

- `03_refactor_strategy_to_unified_reference.md`
  Предлагаемая целевая архитектура: что сохранить, что очистить, что вынести в новый справочник, как мигрировать без поломки бизнес-процессов.

Ключевая мысль:

- текущий `OEM` слой нельзя просто удалить или "почистить до пустоты";
- в нем уже живут важные рабочие сущности: применяемость, BOM, альтернативы, материалы, supplier links, machine-specific overrides;
- переделывать нужно не "вместо OEM", а "над OEM и рядом с OEM", оставляя его operational layer.
