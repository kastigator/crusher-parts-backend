## 2. Текущая сквозная модель процесса

На текущий момент процесс понимается так:

`Заявка клиента -> RFQ -> Покрытие -> Сценарий -> Логистика -> Экономика -> Selection -> Sales Quote -> Quote Revision -> Contract -> PO -> Исполнение -> Supplier Quality`

И дальше:

`Supplier Quality -> Reliability / Risk -> следующие RFQ`

## 3. Что уже доведено

### 3.1. RFQ / Logistics / Economics

- `Incoterms` и `incoterms_place` снапшотятся в `rfq_coverage_option_lines`.
- `origin_country` больше не подменяется `supplier_country` в customs/economics.
- warnings по логистике и экономике усилены.
- FX-провайдер обновлен с нормальной поддержкой основных валют, включая `RUB`.
- `Selection` стал execution baseline, а не просто финансовым слепком:
  - `route_type`
  - `incoterms`
  - `incoterms_place`
  - `lead_time_days`
  снапшотятся в `selection_lines`.

### 3.2. Seller-side

- Продавец работает через `sales_quote_revisions`.
- В `sales_quote_lines` есть `line_status`.
- Запрещено добавлять в rev КП строки вне исходной `client_request_revision`.
- Поставщики на seller-side маскируются через `public_code`, а в `selection_lines` сохраняется `supplier_public_code_snapshot`.

### 3.3. Contract / PO

- Для одной клиентской заявки допускается только один финальный `signed`-контракт.
- Контракт привязан к конкретной `sales_quote_revision`.
- `PO` создается по подписанной rev КП, а не по всему исходному `selection`.
- `PO` автоподхватывает baseline-поля из `selection_lines`.
- Первый `PO` переводит контракт из `signed` в `in_execution`.

### 3.4. Contract execution lifecycle

Сейчас в backend и UI уже введена рабочая модель статусов:

- `draft`
- `sent_to_client`
- `signed`
- `in_execution`
- `completed`
- `closed_with_issues`

Условия закрытия:

- `completed` допускается только если:
  - есть `PO`;
  - все `PO` по scope подтверждены;
  - нет открытых `supplier_quality_events`.
- если исполнение закончено, но остались отклонения, используется `closed_with_issues`.

### 3.5. Supplier Quality

- События качества создаются не только из карточки поставщика, но и из `PO`.
- Событие тянет трассировку до:
  - `selection`
  - `selection_line`
  - `sales_quote`
  - `sales_quote_line`
  - `oem_part`
- События качества можно редактировать и закрывать.
- Supplier quality теперь автоматически влияет на:
  - `part_suppliers.reliability_rating`
  - `part_suppliers.risk_level`
- Автопересчет пишется в `supplier_risk_overrides` с `source = QUALITY_EVENTS`.

### 3.6. Supplier quality в decision layer

Supplier quality уже показан:

- в карточке поставщика;
- в supplier quality summary;
- в RFQ `Coverage`;
- в RFQ `Scenarios`.

То есть закупщик уже видит reliability/risk в момент выбора поставщика, а не только в каталоге.

## 4. Бизнес-решения, которые уже приняты

### 4.1. Один финальный контракт

По одной клиентской заявке должен быть один финальный контракт.

Следствие:

- несколько `signed`-контрактов на один flow считаются ошибочным/нецелевым состоянием;
- закупка ориентируется на один финальный коммерческий baseline.

### 4.2. Supplier quality должен влиять на supplier scoring

Supplier quality не является просто журналом инцидентов.

Это инструмент накопления статистики надежности поставщика для будущих RFQ.

### 4.3. Отдельный approved procurement handoff пока не вводим

Пока можно жить без отдельной сущности `approved procurement handoff`, потому что роль handoff сейчас уже выполняет связка:

- финальный контракт;
- конкретная `sales_quote_revision`;
- `PO` logic, которая берет только approved active lines.

Если позже понадобится сложный release management, partial execution или формальный handoff screen, к этой идее можно вернуться.
