# Аудит и ТЗ: подмена OEM-каталожного номера своим номером / номером клиента / номером по чертежу

Дата: 2026-03-22  
Статус: аудит выполнен, требуется отдельная реализация  
Источник фактов: кодовая база + прямая проверка живой Cloud SQL базы данных

## 1. Зачем нужен этот функционал

В системе нужен отдельный, явный сценарий, когда:

- мы не хотим показывать поставщику исходный OEM-каталожный номер;
- у нас есть свой внутренний номер;
- у нас есть номер клиента;
- у нас есть номер по чертежу / КД;
- мы хотим, чтобы в RFQ, Excel, DOCX и supplier-side документах уходил не OEM-номер, а наш управляемый заменяющий код.

Важно: это не то же самое, что:

- `Альтернативные оригиналы` между OEM-производителями;
- `Подмена` через альтернативную OEM-деталь в структуре RFQ;
- `public_code` поставщика;
- просто `has_drawing`.

## 2. Что проверено по текущей системе

Проверка выполнена напрямую в живой БД через локальный доступ к Cloud SQL.

### 2.1. Что есть сейчас в данных

В живой БД подтверждено наличие строк заявок с заполненным `client_part_number`:

- `client_request_revision_items.id = 261`, `client_part_number = 999`
- `client_request_revision_items.id = 260`, `client_part_number = 02-000-176-2025`
- `client_request_revision_items.id = 255`, `client_part_number = 195-27-33111`
- `client_request_revision_items.id = 254`, `client_part_number = 195-27-33111`
- `client_request_revision_items.id = 253`, `client_part_number = 17-403-100-801`

Вывод:

- в системе уже есть рабочий слой `client_part_number`;
- этот слой фактически используется как “номер клиента / наш номер” на уровне строки заявки;
- данные не потеряны.

### 2.2. Что есть сейчас в коде

Во frontend и backend есть много ссылок на `client_part_number`, в том числе:

- `ClientRequestsPage.jsx`
- `RfqWorkspacePage.jsx`
- `CoverageTabContent.jsx`
- `ScenariosTabContent.jsx`
- `SelectionTabContent.jsx`
- `EconomicsTabContent.jsx`
- `contracts.js`
- `purchaseOrders.js`
- `salesQuotes.js`

Особенно важная находка:

в `ClientRequestsPage.jsx` есть логика, которая строит строку заявки через:

- `overrides.cat_number`
- `overrides.client_part_number`

Это означает, что концепция “подмены номера” уже была частично заложена в систему, но не доведена до законченного пользовательского сценария.

### 2.3. Что есть сейчас, но это не тот функционал

#### А. Подмена через альтернативную OEM-деталь

Сейчас в RFQ есть `подмена`, но она реализована через выбор другой OEM-детали:

- `rfq_supplier_line_selections.alt_oem_part_id`
- `selected_alt_original_part_id`

Это полезно, но это не свободная подмена номера своим кодом.

#### Б. Таблица `oem_part_unit_overrides`

Проверка живой БД:

- таблица существует;
- записей сейчас `0`;
- поля содержат:
  - `oem_part_id`
  - `client_equipment_unit_id`
  - `status`
  - `replacement_oem_part_id`
  - `note`
  - `effective_from`
  - `effective_to`

Вывод:

- таблица сейчас описывает применимость / замену детали на уровне единицы оборудования;
- но в ней нет полей вида:
  - `client_part_number`
  - `supplier_visible_part_number`
  - `drawing_part_number`
  - `supplier_visible_description`

Значит нужный функционал там сейчас не реализован.

#### В. `supplier_part_aliases`

Таблица существует, но в живой БД записей сейчас нет.  
Сценарий алиасов поставщика не решает задачу “какой номер показывать поставщику в RFQ”.

## 3. Главный вывод аудита

Сейчас система содержит только фрагменты нужного сценария:

1. `client_part_number` на уровне строки заявки;
2. частичную идею `overrides.cat_number`;
3. RFQ-подмену через альтернативную OEM-деталь.

Но законченной, явной бизнес-функции:

> “использовать свой / клиентский / чертежный номер вместо OEM-номера в закупочном и supplier-side контуре”

в системе сейчас нет.

## 4. Какая бизнес-модель нужна

Нужна отдельная сущность или слой данных, который отвечает не за “какая это OEM-деталь”, а за:

> какой номер и какое описание должны видеть внешние участники процесса

В системе должны появиться как минимум три разных понятия:

1. `OEM номер`
- исходный номер производителя;

2. `Номер клиента / наш номер`
- код, по которому мы ведем позицию в своей работе;

3. `Номер для поставщика`
- то, что реально попадет в RFQ, Excel, DOCX, заказ поставщику и supplier-side preview.

Эти значения могут совпадать, но не обязаны.

## 5. Предлагаемая целевая модель

### Вариант, который считаю правильным

Сделать отдельный слой override на уровне:

- `OEM деталь + единица оборудования клиента`
или
- `строка заявки`

лучше первый вариант как master-data, второй как snapshot/override на конкретный кейс.

### 5.1. Master-data слой

Нужна таблица, например:

`oem_part_unit_number_overrides`

Поля:

- `id`
- `oem_part_id`
- `client_equipment_unit_id`
- `is_active`
- `internal_part_number`
- `client_part_number`
- `supplier_visible_part_number`
- `supplier_visible_description`
- `source_type` (`oem`, `client`, `internal`, `drawing`, `manual`)
- `drawing_code`
- `note`
- `created_by_user_id`
- `created_at`
- `updated_at`

Смысл:

- для конкретной OEM-детали на конкретной машине клиента можно зафиксировать, каким кодом мы ее называем внутри и каким кодом показываем наружу.

### 5.2. Snapshot на строке заявки

Когда создается `client_request_revision_item`, в него должны попадать snapshot-поля:

- `client_part_number`
- `supplier_visible_part_number`
- `supplier_visible_description`
- `number_source_type`

Почему snapshot нужен:

- историчность;
- после изменения master-data старые кейсы не должны переписываться.

## 6. Минимально необходимая доработка БД

### 6.1. Новая таблица

Добавить таблицу наподобие:

`oem_part_unit_number_overrides`

### 6.2. Новые поля в `client_request_revision_items`

Добавить:

- `supplier_visible_part_number`
- `supplier_visible_description`
- `number_source_type`

Опционально:

- `drawing_code_snapshot`

### 6.3. Новые поля в downstream snapshot

Чтобы supplier-side документы были стабильны, snapshot желательно протянуть дальше:

- в `rfq_items`
- либо в `selection_lines`
- либо вычислять стабильно от `client_request_revision_items`

Минимум:

- `rfq`-экспорт поставщику,
- контракты,
- заказы поставщикам

должны использовать единый источник supplier-visible номера.

## 7. Как это должно работать в UI

### 7.1. В карточке OEM-детали

Не надо смешивать с `Альтернативными оригиналами`.

Нужен отдельный блок:

`Номера и видимость`

Поля:

- OEM номер
- Наш внутренний номер
- Номер клиента
- Номер для поставщика
- Описание для поставщика
- Есть чертеж / код чертежа

### 7.2. В карточке оборудования клиента

На уровне конкретной единицы техники должна быть возможность сказать:

- для этой машины эта OEM-деталь у нас идет под другим номером;
- и поставщику нужно показывать вот такой номер.

То есть это не просто свойство OEM-детали вообще, а свойство “деталь в контексте конкретной машины клиента”.

### 7.3. В заявке клиента

При добавлении позиции:

- если нашелся override для этой машины, поля подставляются автоматически;
- менеджер может вручную скорректировать supplier-visible номер.

### 7.4. В RFQ

Во всех supplier-facing местах нужно показывать:

1. `supplier_visible_part_number`, если он задан;
2. иначе `client_part_number`;
3. иначе `OEM номер`.

Именно в таком порядке.

### 7.5. В supplier-side Excel / DOCX / preview

Поставщик должен видеть:

- `supplier_visible_part_number`
- `supplier_visible_description`

и не обязательно видеть исходный OEM номер.

## 8. Где это должно влиять на процесс

Функционал должен работать сквозным образом:

1. `Клиент -> заявка`
2. `RFQ`
3. `Excel поставщику`
4. `Ответы поставщиков`
5. `Покрытие`
6. `Сценарии`
7. `Выбор`
8. `КП`
9. `Контракт`
10. `Заказ поставщику`

Важно:

- supplier-side контур использует supplier-visible номер;
- seller-side и client-side могут использовать client/internal number;
- OEM номер должен оставаться доступен внутри системы, но не быть единственным видимым номером.

## 9. Что нельзя делать

Нельзя пытаться решить эту задачу через:

- `Альтернативные оригиналы`;
- `alt_oem_part_id`;
- `supplier_part_aliases`;
- `has_drawing`;
- случайное использование `client_part_number` без явной модели источника.

Иначе снова получится полуфункциональная, неочевидная схема.

## 11. Что обсуждать в следующем чате

В следующем обсуждении нужно принять 3 решения:

1. Где главный master-data источник:
- на уровне OEM-детали,
- на уровне машины клиента,
- или комбинированно.

2. Какие именно номера нужны:
- наш внутренний,
- клиентский,
- supplier-visible,
- drawing code.

3. Что именно должно видеть:
- клиент,
- продавец,
- закупщик,
- поставщик.

## 12. Итог

Прямая проверка живой БД показала:

- исторические записи `client_part_number` в системе есть;
- полноценной законченной функции подмены OEM-номера своим supplier-visible кодом сейчас нет;
- текущая логика разорвана между заявкой, частичными overrides и RFQ-подменой через другую OEM-деталь.

Нужна отдельная, ясная модель:

**“какой номер система хранит как OEM, какой номер мы используем внутри, и какой номер мы показываем поставщику”**

Без этого функционал и дальше будет восприниматься как потерянный или недоделанный.
