# Аудит доменной карты для ИИ-агента

Дата: 2026-05-17

Цель документа: уйти от настройки агента точечными правками под каждый пример запроса и собрать системную карту: как пользователь говорит, что это значит в интерфейсе, какие backend-инструменты и сущности должны использоваться, где есть исторические расхождения имен.

## Главный вывод

Прямой доступ агента к базе сам по себе не решит проблему понимания. Он даст больше данных, но не даст бизнес-смысла. Правильный слой для агента:

1. Пользовательские термины и синонимы.
2. Канонические бизнес-сущности.
3. Безопасные backend-инструменты чтения.
4. Черновики действий на изменение данных.
5. Выполнение изменений только после подтверждения пользователя через штатные endpoints.

То есть агент должен работать не как SQL-консоль, а как помощник поверх системы.

## Текущие слои системы

### Интерфейс

Основные разделы, которые видит пользователь:

- Client Request Workspace: рабочий стол по заявкам, назначения, RFQ в работе.
- RFQ Workspace: закупочный процесс, поставщики, ответы, покрытие, экономика, выбор, заказы поставщикам.
- Клиенты: карточки клиентов, контакты, адреса, банки, техника клиента, история заявок и контрактов.
- Поставщики: карточки поставщиков, контакты, адреса, качество, заказы.
- Детали поставщиков: артикулы поставщиков, цены, вес/габариты, связи с OEM и standard parts.
- OEM детали: оригинальные детали, применяемость, BOM, документы, материалы, ТН ВЭД, вес/габариты.
- Стандартные детали: канонический каталог типовых изделий, например болты, гайки, крепеж, электродвигатели.
- Классификатор оборудования: дерево оборудования, узлов и моделей техники.
- Материалы: справочник материалов.
- Коды ТН ВЭД: таможенные коды, описания и пошлины.
- Единицы измерения: допустимые единицы и использование кодов единиц.
- Обзор и качество: очереди нормализации каталогов.
- Пользователи и роли: права, роли, активность пользователей.
- Корзина: восстановление удаленных записей.

### Backend

Backend уже содержит часть безопасных инструментов для агента:

- карта системы;
- бизнес-процесс;
- политика действий;
- доменный словарь;
- универсальный поиск по основным сущностям;
- таймлайн клиента;
- таймлайн RFQ от заявки до ответов поставщиков, выбора закупки, КП, контракта и PO;
- сводка качества каталогов и детальные очереди нормализации;
- объяснение использования единиц измерения;
- список кодов ТН ВЭД по пошлине;
- поиск кандидатов на привязку ТН ВЭД;
- поиск и чтение документов из GCS для OEM и RFQ.

Этого достаточно для первого пилота, но недостаточно для стабильного агента по всей системе.

## Расхождения имен

### OEM детали

Пользователь: OEM деталь, оригинальная деталь, оригиналка, каталожный номер.

Frontend: `/original-parts`.

Backend/API: маршрут совместимости `/original-parts`, но смысловая сущность OEM.

База: `oem_parts`.

Правило для агента: всегда говорить “OEM деталь”, не выводить пользователю `original_parts` или `oem_parts`, если он сам не просит технические детали.

### Заявки клиентов

Пользователь: заявка, заявка клиента, запрос клиента, потребность клиента, заказал клиент.

Frontend: Client Request Workspace, `/client-requests`.

База: `client_requests`.

Legacy-след: `client_orders`.

Правило: говорить “заявка клиента”. Старое `client_orders` считать только техническим alias.

### Ответы поставщиков

Пользователь: ответ поставщика, предложение поставщика, ответ на RFQ.

Frontend: RFQ Workspace, вкладка ответов.

База: `rfq_supplier_responses`.

Опасная ошибка: `supplier_responses` уже приводило к падению агента.

Правило: в инструментах использовать только `rfq_supplier_responses`, пользователю говорить “ответ поставщика на RFQ”.

### Контракты

Пользователь: контракт, договор, незакрытый договор.

Frontend: `/contracts`.

База: `client_contracts`.

Правило: для пользователя “контракт/договор с клиентом”, технические имена скрывать.

### Заказы поставщикам

Пользователь: PO, заказ поставщику, покупной заказ.

Frontend: `/purchase-orders`.

База: `supplier_purchase_orders`.

Правило: не путать с заявкой клиента. Это финальная закупочная стадия после выбора/КП/контракта.

### Единицы измерения

Пользователь: ед. изм., шт, кг, см, единицы.

База: `measurement_units`, но многие рабочие поля пока хранят строковый код единицы.

Правило: агент должен объяснять “справочник задает допустимые коды и показывает использование”, а не говорить о колонках `uom`.

## Пользовательские намерения и правильный маршрут агента

### Найти объект

Примеры:

- “найди удоканскую медь”
- “есть такой поставщик?”
- “найди деталь 12345”
- “какой материал 40Х?”

Маршрут:

1. Определить возможный тип: клиент, поставщик, OEM, деталь поставщика, standard part, материал, ТН ВЭД.
2. Если тип неясен, искать по всем основным сущностям.
3. Вернуть человеческий список: тип, название, раздел интерфейса.
4. Не показывать raw route и id как основной ответ.

Текущий инструмент: `search_business_objects`.

Недостаток: нет поиска по некоторым рабочим сущностям, например RFQ, КП, PO, пользовательская активность.

### Проверить историю клиента

Примеры:

- “Удоканская медь что-нибудь заказывала?”
- “какие по этому клиенту контракты?”
- “что было по клиенту?”

Маршрут:

1. Найти клиента по названию.
2. Построить таймлайн: заявки, RFQ, КП, контракты, заказы поставщикам.
3. Ответить результатом, а не только найденной карточкой клиента.

Текущий инструмент: `get_business_object_timeline` для клиента.

Недостаток: аналогичные таймлайны нужны для поставщика, OEM детали, детали поставщика, RFQ.

### Коды ТН ВЭД и пошлина

Примеры:

- “какие коды имеют пошлину 10%?”
- “подбери ТН ВЭД для поршня”
- “привяжи этот код к таким каталожным номерам”

Маршрут:

1. Если указан процент пошлины, использовать фильтр по ставке.
2. Если указан товар или описание, искать по коду/описанию.
3. Если пользователь просит привязать, подготовить черновик связей.
4. Изменение выполнять только после подтверждения.

Текущие инструменты: `list_tnved_codes_by_duty_rate`, `find_tnved_assignment_candidates`.

Недостаток: нужен общий инструмент “подбор кода по описанию детали” с понятным ответом и кандидатами.

### Единицы измерения

Примеры:

- “где используется кг?”
- “почему pcs есть в системе?”
- “какие единицы измерения есть?”
- “как использовать созданную единицу?”

Маршрут:

1. Показать единицу как бизнес-справочник: код, название, тип, базовая единица, коэффициент.
2. Показать использование по разделам: OEM, детали поставщиков, RFQ, заявки, материалы, standard part.
3. Для новых единиц объяснять: создание добавляет допустимый код, затем этот код выбирается в формах и сохраняется в рабочих сущностях.

Текущий UI уже показывает использование, но tooltip нечитаем и содержит технические источники.

Текущий инструмент агента: `explain_measurement_unit_usage`. Он должен отвечать человечески: где единица используется в разделах системы и как ее выбирать в формах, без показа технических полей.

### Качество каталогов

Примеры:

- “что не заполнено по каталогам?”
- “где нет веса и габаритов?”
- “что надо нормализовать?”
- “какие OEM не связаны со standard parts?”

Маршрут:

1. Получить сводку качества.
2. Разложить по очередям: классификация, standard part, вес/габариты, единицы, ТН ВЭД.
3. Предложить порядок исправления.

Текущий инструмент: `get_catalog_quality_queue`.

Уже покрыты ключевые очереди: OEM без связи со standard parts, OEM без полного веса/габаритов, детали поставщиков без связи со standard parts, детали поставщиков без веса/габаритов, классы standard parts без настроенных полей, standard parts без связей. По мере появления новых правил нормализации очереди нужно расширять.

### RFQ и закупочный процесс

Примеры:

- “что происходит по RFQ-13?”
- “кто из поставщиков ответил?”
- “почему выбран этот поставщик?”
- “что передано продавцу?”

Маршрут:

1. Найти RFQ.
2. Показать состав заявки и позиции.
3. Показать поставщиков, отправки, ответы, покрытие.
4. Показать выбранный сценарий, экономику, КП, контракт и PO.

Текущий инструмент: `get_rfq_timeline`. Он собирает RFQ как бизнес-цепочку: заявка, позиции, поставщики, ответы, выбор закупки, КП клиенту, контракт и заказы поставщикам.

### Документы и файлы

Примеры:

- “проанализируй PDF заявки”
- “посмотри документ в карточке OEM”
- “из Excel поставщика создай черновик ответа”
- “найди в файле клиента позиции”

Маршрут:

1. Если файл загружен в чат, извлечь текст/таблицы/изображение.
2. Если файл уже в системе, найти документ через GCS-связи.
3. Сопоставить найденные сущности с клиентами, OEM, supplier parts, ТН ВЭД.
4. Предложить черновик действий.
5. Запись только после подтверждения.

Текущий статус: загрузка файлов в чат есть; GCS поддержан для OEM документов и RFQ документов.

Недостаток: не покрыты все типы документов, например документы контрактов/PO, если они хранятся иначе.

## Нужная архитектура агента

### 1. Доменный словарь

Должен хранить:

- пользовательские слова;
- название в интерфейсе;
- canonical entity;
- legacy technical aliases;
- разрешенные read-tools;
- разрешенные write-actions;
- что нельзя показывать пользователю.

### 2. Матрица намерений

Должна отвечать на вопрос: если пользователь сказал X, какие инструменты агент обязан использовать?

Например:

- “заказывал клиент?” -> таймлайн клиента.
- “пошлина 10%” -> фильтр ТН ВЭД по ставке.
- “нет веса” -> очередь качества логистики.
- “привяжи код” -> черновик привязки + подтверждение.
- “что с RFQ” -> таймлайн RFQ.

### 3. Безопасные инструменты чтения

Минимально нужны:

- `search_business_objects` - общий поиск.
- `get_client_timeline` - уже частично есть.
- `get_rfq_timeline` - уже есть.
- `get_catalog_quality_queue` - уже есть.
- `explain_measurement_unit_usage` - уже есть.
- `get_supplier_timeline`.
- `get_oem_part_timeline`.
- `get_supplier_part_timeline`.
- `search_tnved_codes`.
- `list_system_documents`.
- `read_system_document`.

### 4. Подтверждаемые действия

Для изменения данных агент должен формировать не SQL, а план:

- действие;
- найденный объект;
- поля до/после;
- риск;
- какой штатный endpoint будет использован;
- что попадет в корзину или аудит.

После подтверждения пользователь нажимает “Выполнить” или пишет подтверждение.

### 5. Запрет прямого SQL для записи

Прямой SQL допустим только для внутренней диагностики разработчика. Для агента в интерфейсе лучше не давать произвольный SQL, потому что:

- он обходит права интерфейса;
- может не создать запись корзины;
- может не создать activity log;
- может нарушить связи;
- может показать пользователю технические детали.

## Что сделать дальше

### Шаг 1. Закрепить словарь и матрицу намерений в backend

Базовая матрица намерений уже добавлена в `aiAgentDomainContext.js`: поиск объектов, история клиента, ТН ВЭД, качество каталогов, единицы измерения, RFQ, документы и подтверждаемые действия. Ее нужно расширять по мере тестирования реальными пользователями.

### Шаг 2. Добавить недостающие read-tools

Приоритет:

1. `get_supplier_timeline`.
2. `get_oem_part_timeline`.
3. `get_supplier_part_timeline`.
4. `search_tnved_codes` по описанию товара.
5. `draft_agent_action`.

### Шаг 3. Добавить экран “предлагаемые действия агента”

Пока агент не должен сам создавать/менять записи. Но он должен уметь показывать структурированный черновик:

- что найдено;
- что он предлагает создать/изменить/связать;
- какие данные не уверены;
- что пользователь должен подтвердить.

### Шаг 4. Собрать тестовый набор запросов

Нужен файл с 50-80 типовыми пользовательскими фразами:

- кривые названия клиентов;
- вопросы по ТН ВЭД;
- вопросы по весу/габаритам;
- вопросы по RFQ;
- вопросы по контрагентам;
- загрузка PDF/Excel;
- просьбы создать/привязать/удалить.

После каждого изменения агента прогонять этот набор и смотреть, не начал ли агент опять говорить техническим языком.

## Вывод для настройки агента

Агент можно сделать полезным уже сейчас, на стадии тестирования системы. Но он должен быть настроен как бизнес-помощник, а не как универсальный доступ к базе. Лучший путь:

1. Расширить доменную карту.
2. Добавлять read-tools по крупным бизнес-сценариям.
3. Все изменения делать через подтверждаемые планы.
4. Постепенно учить агента новым разделам через обновление словаря, playbook и инструментов.

Такой подход масштабируется: когда в системе появится новый раздел, мы не “переобучаем модель”, а добавляем в backend доменный объект, пользовательские синонимы, read-tool и правила действия.
